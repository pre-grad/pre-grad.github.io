Stop Rules

If a task does not directly improve CV, cert, deployment, or applications — it waits.
Hard stops
AWS deployment eating 5+ days? Ship on Render or Railway. AWS CP keeps the cert. The deployed project doesn't need to be on AWS to matter. A live demo beats a noble corpse.
RDS Postgres migration? Only if everything else is done. Never before.
GitHub Actions / CI/CD? Only if everything else is done.
New cert / second language / pivot? Finish what's on the plan first.
Restart Linux? Skipped. You'll learn the missing chapters from AWS pain.
Restart Git from chapter 1? Resume from chapter 5.
Docker / Go / AI agents / functional programming? Not on critical path. Post-job.
Polishing one section for days? Plan rewards shipping.
AI writing your coding reps? The rep doesn't count.
Replanning every few days? Execute, don't redesign.
The 10 rules (re-read weekly)
1.
No more re-planning. Execute what's written.
2.
AI is a tool, not a ghostwriter. Debug with AI; don't let AI type your code.
3.
Perfect is the enemy of done. SQLite live > Postgres half-deployed.
4.
The dissertation is your biggest asset. Lead with it.
5.
Dubai address on everything. Karachi off the CV.
6.
One no-AI coding rep per day. Non-negotiable.
7.
Track applications. Count them or you can't improve.
8.
When stuck, ship anyway. Momentum > polish.
9.
Cloud-leaning backend is the entry strategy, not your identity. Re-angle if backend is silent by late July.
10.
The deployment escape hatch. Render/Railway after 5+ days of AWS pain.

Theory

🎯 Core methodology — Execution > Re-planning
The plan is treated like a mission plan, not a study timetable.
Critical path first. Only tasks that improve employability matter: CV, cert, deployment, applications, coding fluency.
Daily minimum wins. Every day produces something visible.
No over-planning. Planning is limited. Execution is the job.
Proof over intention. A deployed repo, passed cert, submitted application counts. "I thought about it" doesn't.
Momentum beats perfection. A rough but finished task beats a polished task that never ships.
🧠 Five learning theories the plan uses
Backward planning — start from deadlines: AWS by Jun 1, deploy by Jun 15, apps by Jul 20.
Chunking — big goals broken into weekly + daily tasks.
Active recall — explain dissertation, routes, security choices from memory.
Deliberate practice — Codewars / LeetCode reps, no AI.
Project-based learning — dissertation deployment is the main learning surface for Flask, SQL, Git, AWS, debugging, interviews.
📅 The daily formula
60–90 min   main deliverable (CV / AWS / GitHub / deploy / apps)
30–45 min   coding fluency (boot.dev current module)
15–20 min   daily rep (Codewars / LeetCode, no AI)
20–45 min   dissertation memory (re-read, summarise, explain)
5 min   log progress
